# PLAN.md - CIP3 喚醒提示音（Flash 內嵌 + 喚醒後播放）規劃

## 1. 目標與背景
在 ESP32 偵測到喚醒詞 `heymiaomiao` 後，播放一段本地短提示音（ack tone），再進入原本 3 秒錄音並傳送至 Server 的流程。

此項目符合 SPEC.md 的邊緣端責任：
- ESP32 只做喚醒、錄音、回饋播放
- 控制決策與 MQTT Trigger 仍由 Server 端負責

## 2. 非目標（本階段不做）
- 不更動 Server ASR/LLM/MQTT 決策流程
- 不新增雲端音效下載或動態更新機制
- 不做複雜音效管理（多語音包、混音、音量自動化）

## 3. 技術策略（規劃版）
- 音訊資源：將短音效檔以 `EMBED_FILES` 內嵌於 firmware flash
- 音檔格式優先：`PCM 16kHz / 16-bit / mono`（可選 WAV 包裝）
- 執行流程：`wake detected -> 播放提示音 -> cooldown -> 錄音3秒 -> 上傳`
- 防重複喚醒：加入 suppression window（建議 1200ms，可調）
- I2S 使用策略：先採「序列式」RX→TX，避免全雙工複雜度

## 4. 前置條件與決策 Gate
- 硬體：MAX98357A 到貨並確認接線（BCLK/LRCK/DIN）
- Pin 規劃：確認是否與 INMP441 共享時脈腳位
- 音效長度定版：建議 150~600ms，預設 500ms
- 量測工具可用：`idf.py monitor` 可觀察喚醒、播放、錄音時序

## 5. 資源評估（基於目前 size）
- 目前 image 約 `965,772 bytes`
- 提示音容量預估（16k/16-bit/mono）：
  - 100ms：約 3.2KB
  - 200ms：約 6.4KB
  - 300ms：約 9.6KB
- 結論：單一或少量提示音對目前 Flash 預算風險低

## 6. 風險與緩解
- 風險A：提示音被麥克風收進後續錄音
  - 緩解：播放後加入 cooldown（100~200ms）再啟動錄音
- 風險B：提示音觸發二次喚醒
  - 緩解：喚醒後 suppression window + 暫停喚醒判定
- 風險C：I2S TX/RX 切換導致不穩
  - 緩解：先做單執行緒序列切換，確保狀態明確

## 7. 驗收定義（完成條件）
- 喚醒成功後有可聽見提示音（硬體條件成立）
- 提示音後可穩定錄滿 3 秒並成功傳輸
- 不出現明顯重複喚醒與流程卡死
- 不影響既有 WebSocket 音訊傳輸與 Server 端處理

## 8. 實施時程（暫定）
- Day 1：硬體接線確認 + I2S TX 初始化驗證
- Day 2：嵌入音檔 + 播放函式 + 狀態機串接
- Day 3：時序調參（cooldown/suppression）+ 回歸測試

## 9. 目前狀態
- 本文件為「先規劃、不動工」版本
- 待 MAX98357A 到貨後再進入開發與測試執行
